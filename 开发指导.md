## 一、整体定位与设计原则

在动手写代码前先把几件事讲清楚,避免开发中反复返工:

**1. 论文组件 ↔ 代码模块严格一一对应**。4.3 节每个小节对应一个独立目录/文件,答辩时可以直接指着代码说"这就是 4.3.5"。不要把多个组件塞进一个大文件。

**2. 会诊协调器是唯一的"大脑",其他模块都是被它调用的工具**。路由调度器、专家智能体、意见综合、安全审查都不准互相直接调用;不准直接读写数据库(数据库只由协调器管)。这一条来自 4.3.1,违反它会让状态一致性崩盘。

**3. 模型层与系统层严格隔离**。`app/model/` 只提供一个 `generate(prompt, rank_hint=None)` 接口,对外屏蔽 MoE-LoRA 的内部机制;`app/agents/` 只管构造 prompt、解析输出,不碰模型内部。这样做的好处是:如果你后续要做消融(比如换基座模型),只需改 `app/model/`。

**4. 一次会话 = 一个病例**。来自数据库设计文档,这个约束会极大简化状态管理。不要在同一个 session 里支持"中途换病人"。

**5. 两层专家是软对应,不是强绑定**。系统层科室智能体只通过 prompt 注入科室名和推理重点;模型层各 MoELoRALinear 的 `layer_router` 完全由输入隐状态自行决定激活权重。代码里**严禁**出现"根据科室名 X 强制激活第 Y 号专家"的分支。这是真论文第四章的核心亮点,写错了整章亮点坍塌。

## 四、关键实现提示

这几处论文里定义清楚但容易实现错,提前标红:

**(1) 路由调度器的置信度计算**(4.3.2)。先按科室标签对命中片段分组,取该科室下相似度**最大值**(不是均值!),再对所有科室原始分数做 softmax。这个细节论文正文写得很明确,是避免低相关片段累积拉高噪声科室。

**(2) 一致性检验用余弦最小值**(4.3.5 第一级)。不是均值,因为医疗场景对冲突容忍度低——只要有一对意见显著偏离就要进下一级。

**(3) 第二级加权中的 `relevance`、`confidence` 直接从 `RoutingResult` 读,不要重算**。这两项在路由阶段已存好在 `inference_meta` 里,重算会浪费资源还可能不一致。`completeness` 是对 `DepartmentOpinion` 五字段的覆盖度评分(每缺一字段线性扣分)。

**(4) 第三级仲裁:`rank_hint` 由协调器传入,不是 aggregator 自主决定**(4.3.5)。真论文原文是"协调器通过动态秩控制模块将仲裁推理的秩配置提升至更高档位",动作发起方必须是协调器。代码上用 `coordinator_hook` 回调——aggregator 判断命中 L3 时回调协调器的 `_arbitrate`,由协调器调用 `model.generate(prompt, rank_hint=max(rank_bins))` 获得仲裁稿,再返回给 aggregator 的 `level3_arbitration.py` 组装最终报告。**这条链路是论文的"系统层与模型层协同"核心亮点,代码里必须有显式的 `rank_hint=max(...)` 参数,否则亮点落不了地。**

**(5) 并行模式用 `asyncio.gather` + `asyncio.wait_for` 组合**(4.3.4 + 4.3.1 超时容错)。每个 agent 独立设置超时阈值,超时一次 → 重试一次 → 标 `absent` → 综合阶段跳过。不要用线程池,会阻塞 FastAPI 事件循环。

**(6) 安全审查的降级不是拒绝**(4.3.6)。替换风险表述 + 保留非风险内容结构 + 追加强化免责声明,三步都要做。免责声明用固定模板字符串,不走 LLM,避免引入新的语义风险。

**(7) `inference_meta` 要存全**:各科室置信度、会诊模式、各 agent 耗时、rank 档位、层内路由权重摘要。这些数据是 4.5.6 案例分析和专家分化可视化的唯一数据源,会诊时不存后面补不出来。

**(8) 两层专家是软对应**:`DepartmentAgent.analyze` 的 prompt 里只允许出现科室名和推理重点,绝不能有"请激活第 X 号专家"这类指令。模型层 `MoELoRALinear.layer_router` 的权重完全由输入隐状态决定,不接受外部科室标签控制。这是真论文第四章的核心亮点,违反它整章设计坍塌。

## 六、几个容易踩坑的提醒

**不要把模型加载放到请求链路里**。FastAPI 启动时在 `lifespan` 里加载一次模型,全局共享。每次请求都加载会慢到无法使用。

**FAISS 索引也预加载到内存**。放在应用启动时加载,不要每次查询都读盘。

**异步/同步边界要清楚**。FastAPI 是 async 的,但 HuggingFace `model.generate()` 是同步的,必须 `await asyncio.to_thread(model.generate, ...)` 包一层,否则并行模式实际会退化成串行。这点最容易踩坑,强烈建议阶段 7 接入真实模型时第一时间验证。

**SQLite 在并发写入下会锁**。如果 orchestrator 多实例并发写 message 表,加 `check_same_thread=False` + 互斥锁,或者直接换 PostgreSQL(毕设场景下 SQLite + 单进程够用)。

**阈值 τ、边际 m、权重 α/β/γ 先给默认值,最后在 4.5 节调**。别开发阶段就纠结阈值取多少,默认 τ=0.3、m=0.2、α=β=γ=1/3 够你跑通,具体数值等 4.5 节用验证集调。

**提示词别硬编码在 .py 里**。全放 YAML,改 prompt 不改代码。答辩老师问"为什么这么设计提示词",你可以现场打开 YAML 指给他看。

**Streamlit 前端调后端建议用同步 requests**,不要在前端写 async——Streamlit 的 rerun 机制与 async 配合容易出坑。后端是 async 就行。
